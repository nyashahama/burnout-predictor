# Action Engine — Habit-Forming Burnout Predictor

**Date:** 2026-04-11
**Status:** Approved

## Problem

Users check in, get a score, and don't know what to do next. The app feels like a thermometer — it tells you the temperature but not whether to grab a jacket. There's no closed feedback loop between "I did X" and "my score improved because of X." Users with unpredictable schedules (night workers, variable days) are poorly served by fixed-time reminders and day-of-week assumptions. The daily habit feels fragile — miss a day and there's no pull to come back.

## Design: The Action Engine

The app transforms from a passive scoreboard into an action coach with a closed daily loop:

```
Check in (2 inputs, 5 sec)
      ↓
"Tell me more?" expand (optional, 30 sec)
      ↓
Action Plan appears (score-tiered)
      ↓
Follow-up surfaces from yesterday's note
      ↓
Next check-in → "Walking outside dropped your score 8 points"
```

Two habit hooks reinforce each other: **streak momentum** (daily pull) and **insight reveals** (curiosity pull). Scheduling adapts to the user's personal rhythm, not the other way around.

## 1. Post-Check-In Action Plan

After submitting a check-in, the dashboard transforms into a personal action plan. Three tiers based on score:

### Score ≤ 40 — "Keep it going"
- Compact card with 1-2 maintenance suggestions
- Shows yesterday's small wins if any were logged ("You walked outside yesterday — keep that up")
- Links to insight reveals if new patterns were detected

### Score 41-64 — "Ease the load"
- 3 actions broken into Tonight / Tomorrow Morning / This Week
- Actions are generated by the existing `buildDynamicRecoveryPlan()` — keyword-aware (notes mentioning "deadline" trigger deadline-specific actions), role-aware, stress-level-aware
- Note context banner: "You mentioned deadlines — these actions are calibrated to that."

### Score ≥ 65 — Full alert + recovery plan
- `BurnoutAlert` banner at top (streak phrase, danger zone context, forecast)
- Full `RecoveryPlan` with checkboxes, progress tracker (X/Y done)
- Three sections: Tonight, Tomorrow, This Week
- Checkbox state persisted per day

### Next-day feedback loop
When a user checks in and their score improved from yesterday:
- "Your score dropped 8 points from yesterday. You logged 'walked outside' — that's your best recovery move so far."
- Uses the existing `findWhatWorksForYou()` combined with the new small wins data
- This is the closed loop: action → result → validation

### Follow-up questions
If a previous note mentioned a future event ("big deadline Wednesday"), the follow-up question surfaces at the top of the next day's check-in:
- "You mentioned a deadline — how did it go?"
- Uses the existing `parseFollowUpSignals()` / `getFollowUpForToday()` code (currently orphaned, never called)
- Follow-up cleared after being shown (`clearFollowUpForToday()`)

### Components to wire in
| Component | Current state | Change |
|---|---|---|
| `BurnoutAlert` | Built, not imported | Import in dashboard, render when score ≥ 65 |
| `RecoveryPlan` | Built, not imported | Import in dashboard, render when score ≥ 65 |

| `parseFollowUpSignals` / `getFollowUpForToday` | Built, not called | Call from CheckIn component, surface follow-up |

## 2. Richer Check-In Flow

### Default (5 seconds)
- Stress level (1-5) — same as now
- Note — same as now

### Expanded section (optional, ~30 seconds)
An expandable "Tell me more?" section below the note field:

- **Energy level** (1-5) — reuses existing `energy_level` column on `check_ins`
- **Focus quality** (1-5) — reuses existing `focus_quality` column
- **Hours worked** — reuses existing `hours_worked` column
- **Small wins:** "Did anything help today?" — free text input for recovery actions (walked, slept early, no meetings, etc.)
- **Physical symptoms** — chip selector (headache, muscle tension, fatigue, trouble sleeping, appetite changes) — reuses existing `physical_symptoms` TEXT[] column

### Guided reflection prompt
Below the expanded section, a single contextual question:
- High-stress days (4-5): "What's one thing you could do tomorrow to lighten the load?"
- Low-stress days (1-2): "What kept you steady today?"
- Medium-stress days (3): "What's on your mind?"

The response gets appended to the note field with a `\n\n---\n` separator so the server stores it as part of the note. When the feedback loop surfaces "what worked / what kept you steady," it extracts the reflection portion from the note using the separator, so it can be displayed separately from the raw note.

### Adaptive language
- The check-in heading says "How are you?" — no "Good morning" or "End of day" assumptions
- Works for any schedule, any time of day

### Database change
- Add `small_wins TEXT` column to `check_ins` table (new migration)
- All other expanded fields already exist in the schema

## 3. Habit Hooks — Streaks & Insight Reveals

### Streak System

#### Visual streak tracker
- **Desktop:** In the sidebar, show current streak number + 7-day dot row (filled = checked in, empty = missed). No day-of-week labels.
- **Mobile:** In the top bar, same compact display.
- Dots represent the last 7 check-in days, not Monday-Sunday.

#### Streak milestones
Dismissable celebration cards at the top of the dashboard:

| Days | Milestone | Message |
|---|---|---|
| 3 | Finding your rhythm | "You're building the habit. Three days of data." |
| 7 | One full week | "Your data is getting meaningful. Patterns are starting to form." |
| 14 | Pattern unlocked | "The app can now see your personal trends." — links to pattern insights |
| 30 | Signature discovered | "Your burnout fingerprint is clear." — links to personal signature |
| 60 | Long arc visible | "You can see your trajectory over months." — links to long arc narrative |

Milestones currently fire to localStorage via `detectRecoveryMilestone()`. Change to store in `insight_metadata` server-side so they appear on every device and survive cache clears.

#### Streak forgiveness
- Miss 1 day: streak **pauses** (doesn't reset). Dot stays empty but streak counter holds.
- Miss 2 consecutive days: streak **resets** to 0.
- Rationale: One slip shouldn't kill a long streak. Two misses suggests the habit is genuinely broken.

### Insight Reveals

New discovery cards in the dashboard, replacing the current static pattern list:

| Trigger | Card |
|---|---|
| 7+ check-ins | **Pattern detected**: "Your Wednesdays are 23% heavier than your average. Meetings day?" |
| 14+ check-ins with small wins | **What works**: "When you walk outside, your next-day score drops an average of 8 points." |
| Any significant score shift | **Score shift**: "Your score moved 12 points since yesterday. Here's why:" with link to the pattern |
| 30+ check-ins | **Personal signature**: "Heaviest on Wed, lightest on Fri. Average: 52. Trend: slowly improving." |

Each card is shown once and dismissable. Dismiss state stored in `dismissed_components` (already exists in DB, already has a `dismissed_at` column and `UpsertDismissedComponent` SQL query).

### Consistency metric
Replace "Danger days" with **consistency**: "You've checked in 18 of the last 21 days (86%)." Frames the habit positively rather than punitively.

## 4. Adaptive Scheduling

### Smart reminders
- No fixed reminder time in onboarding. The app observes check-in times over the first 7 days and sets the reminder to the user's personal median check-in time.
- **Weekly recalibration:** Every week, recalculate the median from the last 14 check-ins. If the pattern shifts (night → morning), the reminder follows.
- **No consistent pattern (std dev > 4 hours):** Drop the fixed reminder. Instead, send a simple "How was today?" at whatever time the user last checked in. One message, one tap.
- **Reminder delivery:** Uses existing Resend email integration and `email_logs` table. The `worker` package already exists. Push notifications are future scope — email ships first.

### Contextual reminder copy
Uses the existing `buildNotificationText()` function, now actually delivered:

| Context | Copy |
|---|---|
| Streak 3+ | "3 days in a row. Keep the data flowing." |
| Missed yesterday | "You skipped yesterday. No judgment — check in when you're ready." |
| Danger zone 3+ days | "You've been in the danger zone for 3 days. Tonight's check-in matters." |
| Score improved | "Your score dropped 8 points yesterday. See what helped." |
| Follow-up event | "You mentioned a deadline yesterday. How did it go?" |

### No time-of-day assumptions
- Dashboard heading: "How are you?" (not "Good morning" / "Good evening")
- Weekly view: "Last 7 check-ins" (not "This week")
- Streak dots: last 7 check-in occasions, not Monday-Sunday
- Reminders: adapted to personal rhythm, not clock-based

### Implementation
- Backend: Add a `NotificationWorker` to the existing `worker` package that queries users needing reminders, computes their median check-in time, and sends contextual emails via Resend
- Store computed reminder times in `user_notification_prefs` (replace the fixed `reminder_time` with adaptive logic)
- Track which reminders have been sent in `email_logs` to prevent duplicates

## 7. API Changes

### `POST /api/checkins` — Extended request body

```json
{
  "stress": 4,
  "note": "Tough day with deadlines",
  "energy_level": 2,
  "focus_quality": 3,
  "hours_worked": 9.5,
  "physical_symptoms": ["headache", "fatigue"],
  "small_wins": "walked outside for 20 minutes"
}
```

All new fields are optional. `stress` and `note` remain the only required fields for the base check-in.

### `GET /api/score` — Added fields in response

```json
{
  "streak": 7,
  "consistency_pct": 86,
  "streak_forgiven": false,
  "has_follow_up": true,
  "follow_up": {
    "question": "You mentioned a deadline — how did it go?",
    "source_date": "2026-04-10"
  }
}
```

### `GET /api/insights` — Added field

```json
{
  "what_worked_today": {
    "action": "walked outside",
    "improvement": 8,
    "evidence": "3 of 4 times you walked outside, your next-day score dropped by an average of 8 points."
  }
}
```
